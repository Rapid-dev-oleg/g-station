# Пайплайн «шаг = скил» + мастер нового расчёта — план

Цель: сделать расчёт **прозрачным и управляемым конвейером шагов**, где каждый
шаг = самодостаточный скил (текст + скрипты), который видно и правят (в т.ч.
через ИИ-помощника), а исполнение идёт **по-шагово в одной сессии агента**
(контекст не теряется). Мастер «Новый расчёт» показывает эти же шаги.

Развилка исполнения решена пользователем: **вариант A — каждый шаг = отдельный
прогон, но в одной сессии Kimi** (см. «Исполнение»).

Макет мастера (шаг 1 «Вход»): артефакт `new-calc-wizard` (кликабельный).

---

## Идея в одном абзаце

Пайплайн типа = упорядоченный список **шаг-скилов** (Вход → Расчёт → Подбор →
Цена → Выход). Один шаг = один скил: **описание** (SKILL.md-текст, инструкции
агенту) + опц. **скрипты/файлы** (формулы, таблицы, прайсы — детерминированная
часть). Шаги исполняются последовательно **в одной сессии агента**, поэтому
шаг N помнит шаги 1..N−1. Между шагами — гейты инженера. Мастер «Новый расчёт»
= те же шаги; шаг 1 «Вход» универсален, но за ним редактируемый скил под тип.

## Два вида шагов (ключевое)

- **LLM-шаг** — описание = инструкции агенту, считает модель (напр. «Подбор»).
- **Скрипт-шаг** — детерминированный, без LLM: формула/таблица/прайс. Напр.
  «DN по таблице расход→DN», «мощность = Q·H/(367·η)», «цена по прайсу CNP».
  Быстро, точно, повторяемо. **Сюда переезжает KNOWLEDGE** (числа, которых
  сейчас в конструкторе нет — только указатели).

Результат скрипт-шага **вливается в сессию сообщением** («DN по таблице = DN65»)
→ следующий LLM-шаг продолжает с этим. Детерминизм + сохранённый контекст.

## Мастер «Новый расчёт»

Шаги мастера = **шаги пайплайна типа** (не отдельные «Тип/Карточка»).

**Шаг 1 «Вход»** (универсален для всех типов, за ним редактируемый скил
`шаг1-вход.md`, где описываем уникальности типа):

- **Тип** станции — селектор; **клиент/проект** — необязательные, привязываются
  в любой момент.
- **Ввод: вручную / из файлов.** Файлы (если загружены) видны на этом шаге.
- **Схема видна ВСЕГДА** (форма из field-spec типа — `DynamicForm`). Вручную
  заполняешь сам; из файлов — ИИ предзаполнил + метки провенанса
  (из ТЗ / допущение / заполнить).
- **Гейт 1** прямо в форме: что извлечено, что додумано (допущение), что пусто.

Дальше «Далее — Расчёт →» → шаг 2 в **той же сессии агента**.

## Редактор шага (скила)

В каждом шаге — доступ к его скилу: **текст (markdown) + файлы/скрипты**, всё
правится. Плюс **🤖 ИИ-помощник**: пишешь «этот параметр считается иначе, вот
правки» → LLM возвращает исправленный файл → **дифф → подтверждение → сохранить**
(с бэкапом версии). Никакой авто-перезаписи скила без предпросмотра.

## Исполнение (вариант A + одна сессия Kimi)

Раннер идёт по шагам:
1. Материализует шаг-скил (описание+файлы) в workspace агента.
2. Собирает вход из состояния («расчётное дело» JSON — уже существует).
3. **LLM-шаг** → прогон агента; **скрипт-шаг** → выполнение кода.
4. Структурный выход вливает в состояние.
5. **Гейт** → пауза на инженера (проверил/поправил/переиграл один шаг).
6. Следующий шаг — **в той же сессии**.

Механика сессии (проверено — CLI умеют):
```
Шаг 1:  kimi --print --output-format stream-json -p "<шаг 1>"  → ловим session_id
Шаг 2:  kimi --print --session <session_id>       -p "<шаг 2>"  → помнит шаг 1
Шаг 3…5: --session <session_id>
```
- Параллельные расчёты (цель — 5 сразу): **явный `--session <ID>` на пайплайн**,
  НЕ `--continue` (иначе сессии перепутаются).
- Контекст растёт от шага к шагу — для 5 шагов ок (это и есть «не теряем»).
- MCP к БД (подбор/цены) доступен на всех шагах сессии.

4. **Скрипт-шаги:** JS-песочница; DN/прайс/формулы → код; KNOWLEDGE переезжает.
5. **Гейты между шагами:** пауза/правка/переигровка шага.
6. **Пилот:** тип целиком через конвейер vs скил, сравнение.

3. **Песочница скриптов:** язык (JS в Node-vm?), насколько свободный код,

   изолированный шаг (чистый вход→выход для тестов) — отдельный прогон без сессии.
