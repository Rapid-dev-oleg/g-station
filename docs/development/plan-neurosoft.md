## Исходные данные клиента (док. июль 2026)

### Дерево систем и объёмы ТКП (среднемесячно, апр–июн 2026)

Периметр — продукция собственного производства с префиксом **G**.

| Тип системы | Серия | ТКП/мес | Приоритет |
|---|---|---|---|
| Пожаротушение | **G-FIRE** | ~12 | 1 (пилот) |
| Хозпит | G-WATER, G-POWER | ~9 | 2 |
| КНС / ЛОС | G-DRAIN, G-CLEAN | ~8 | 3 |
| **Итого** | | **~29** | |

G-FIRE выбран пилотом: наибольший объём, наименьшая вариативность,
жёсткая нормативная база (параметры в таблицах, не в экспертном суждении).
Тираж на остальные типы — после пилота.

### Нормативная база пожарных (жёсткая, из таблиц)

**СП 10.13130.2020**, **СП 8.13130.2009**, **НПБ 88-2001\***.
> Сверить с презентацией (`presentation-fire-station.md` упоминает «СП 8/10/485»)
> — уточнить, актуальны ли СП 485 или НПБ 88-2001*.

Логика ТКП (клиент): **расход → давление → подбор насоса по каталогу →
спецификация → стоимость**.

### Подход к ТЗ расчёта — живой документ

ТЗ формируется **итеративно** в ходе тестирования, а не как статичный документ
на старте: первая версия на реальных запросах → инженеры дают фидбэк по
ошибкам → правим. Это прямо согласуется с задачей #16 (тестирование за неделю)
и с гейтами супервизинга (#12).

## Группировка по направлениям

### A. Ядро расчёта ТКП (LLM-only) — главная ценность
- **A0. Убрать TS-движок** (решение 03.07). Вырезать потребителей `lib/engine`
  из UI/сервера, расчёт целиком через `kimi-calc.ts`. → см. раздел «Немедленно».
- **A1 (#1)** Пожарные станции первыми. *Статус: реализован тип `fire` через Kimi.*
- **A4 (#4)** 5 параллельных расчётов, ~10 мин, масштабируемо. *Основа есть —
  очередь задач; нужно параметризовать лимит параллелизма и нагрузочно проверить.*
- **A12 (#12)** Супервизинг = гейты инженера на промежуточных этапах; по мере
  роста качества гейты схлопывать. *В скиле уже заложены 3 гейта — оформить в UI.*
- **A10 (#10)** СНиП/ГОСТ как доверенные источники: проверка соответствия
  (напр. диаметр насоса по ГОСТ).
- **A11 (#11)** Оценка достоверности внешних источников (1–10, использовать >7),
  приоритет доверенных ресурсов над веб-поиском.
- **A14 (#14)** Веб-поиск оборудования: сперва доверенные источники, потом интернет.
- **A9 (#9)** Управление номенклатурой + добавление производителей и URL их
  каталогов для авто-поиска. *Основа — реестр `SystemType` в БД.*
- **A8 (#8)** Загрузка обучающих материалов пользователями (дообучение/контекст,
  выявление багов).

### B. Платформа и доступ
- **B2 (#2)** Мультикомпанийность + раздельные воркспейсы (мультитенантность;
  внутри — фактически две компании; перспектива продавать продукт другим).
  **Фундамент — заложить рано, до наращивания данных и ролей.**
- **B3 (#3)** Роли: руководитель / инженер / менеджер. *Сейчас в БД только
  `ENGINEER`/`ADMIN` — расширить и развести права.*
- **B5 (#5)** Развернуть на сервере с авторизацией и ролями. *Инфра-деплой
  сделан (сервер VDSina, Docker, автодеплой). Остаётся авторизация+роли+
  доработка кабинета.*

### C. Воронка: CRM и каналы заявок
Каналы/инфраструктура из протокола:
- **C6 (#6)** Битрикс24: все входящие каналы (почта, телефония) → авто-заявка;
  прокладка между Битриксом и каналами для анализа лидов.
- **C7 (#7)** Записи звонков с Билайна → транскрибация (вместо платной услуги Битрикса).
- **C13 (#13)** Телеграм-боты для заявок (фото/голос/файлы → расширенная заявка в Битрикс). Огр.: VPN.
- **C15 (#15)** Дашборд настроек воронки (порог скоринга, квалификация лидов).

Карта AI-точек по стадиям Битрикс24 (из док. клиента) — это целевой охват
направления C. **ТКП (стадия 3) = стык с ядром расчёта (направление A).**

| Стадия | Что автоматизируем (AI) |
|---|---|
| 1. Новая | автоквалификация заявки (тип/объём/регион/бюджет); отсев нецелевых с тендерных площадок; маршрутизация с сайта/писем; автоназначение менеджера; автозаполнение карточки из ЕГРЮЛ/тендерной истории |
| 2. Сбор инфо / выход на ЛПР | автопоиск контактов ЛПР; обогащение карточки (ИНН→отрасль, тендеры, аффилированные); AI-бот сбора техданных у клиента |
| 3. Подготовка КП | **автогенерация ТКП** (подбор оборудования, спецификация, стоимость); актуальные цены/сроки от поставщиков; проверка на нормы (СП/СНиП/ГОСТ); генерация PDF КП |
| 4. КП отправлено / согласование | автоотправка КП; follow-up (день 3/7/14); AI-анализ ответа (согласовано/правки/отказ); автофиксация статуса |
| 5. Согласие / документы | автогенерация договора; пакет (договор, счёт на аванс, спецификация); маршрутизация на согласование; контроль аванса |
| 6. Договор подписан / реализация | задачи производству по авансу; заявка на закупку из спецификации; уведомления о статусе; УПД |
| 7. Отгружено | уведомление об отгрузке; NPS через 2 недели; отчёт по проекту; AI-классификация рекламаций |
| Сквозной (любая стадия) | AI-скоринг сделки; анализ звонков (транскрипция/тональность/договорённости); анализ переписки; контроль активности (брошенные сделки); вероятность закрытия; автоприоритизация портфеля менеджера |

> Объём направления C большой и по сути отдельный продукт поверх Битрикс24.
> Фокус пилота (Фаза 1) — только стадия 3 (ТКП) + минимальный сбор заявок;
> остальные стадии — Фаза 3.

## Решения

### Уже закрыты документом клиента (июль 2026)
- **Приоритет продукта (был вопрос 3):** пилот на **G-FIRE**, затем
  G-WATER/G-POWER, затем G-DRAIN/G-CLEAN. Из воронки в пилот берём только
  стадию 3 (ТКП) + минимальный сбор заявок.
- **Границы с Битрикс24 (был вопрос 5):** Битрикс24 — система воронки; наша
  система встраивает AI по стадиям, генерация ТКП — стадия 3.
- **Подход к ТЗ (был вопрос 4, частично):** живой итеративный документ +
  фидбэк инженеров вместо статичного ТЗ. Метрический регресс-прогон Kimi-пути
  на эталонных кейсах — всё ещё полезен, оформить как часть петли тестирования.

### Открыты — нужны от клиента/команды
1. **Мультитенантность (#2):** уровень изоляции — воркспейс с row-level
   разделением в одной БД или отдельная БД на компанию? Определяет модель данных.
2. **Роли (#3):** матрица прав руководитель/инженер/менеджер (кто что видит и
   может: заявки, расчёты, номенклатура, настройки).
3. **Нормы пожарных:** актуальный перечень — СП 10.13130.2020 + СП 8.13130.2009
   + НПБ 88-2001\*, или СП 485 (расхождение с презентацией)?
4. **Ресурс на воронку:** направление C (7 стадий) — отдельный крупный продукт;
   вести его параллельно ядру или строго после пилота?
