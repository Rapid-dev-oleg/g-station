# План: версионируемая структура типа (TypeVersion) + скилы копией в тип

Статус: черновик, обсуждён с пользователем 23.07.2026. Решения: **единая версия
типа** (TypeVersion снимает всё), **скилы копируются в тип** (эталон
`~/Projects/gidrostroy` не трогаем — read-only), **все изменения (шаги, схемы,
спецификация, карточка, нормы-ссылки) идут через версию**. Пилот — G-FIRE.

## Зачем

Сейчас версионность фрагментирована и скилы общие:
- Схема ввода `TypeSchema` — версионируется (version + draft/active/archived). ✅
- Спецификация `SystemType.specSchema` — плоское поле, без версий. ❌
- Дизайн карточки `SystemType.cardLayout` — плоское поле, без версий. ❌
- Шаги `TypeStep` — без версий (один набор на тип). ❌
- Скилы `шаги/шагN.md` — ОБЩИЕ (эталон, read-only), не по типам; `TypeStep.file`
  = путь в эталоне. ❌ (правка через /methodology правит копию в volume, но она
  общая для всех типов и не версионируется по типу).
- Нормы `Norm` — глобальная библиотека (status active/superseded).

Цель: тип владеет **версионируемым бандлом**; правки — через draft→publish; откат.

## Модель шага (что потребляет/производит) — G-FIRE

| Шаг | Использует | Карточка дизайна (блок) | Выход |
|---|---|---|---|
| 1. Вход | Схема ВВОДА (FieldSpec→DynamicForm) + парсинг + скил шаг1 | Карточка параметров (схема ввода + провенанс) | Q, H, назначение, схема, ограничения |
| 2. Расчёт | Скил шаг2 + НОРМЫ (СП 8/10: расход, продолжительность, V=расход×время) | «Характеристики» (Q/H, резервирование, коллектор, мощность, пуск) | Целевые характеристики |
| 3. Подбор | Скил шаг3 + ИСТОЧНИКИ (БД→API select_pump→доверенные сайты→веб) + схема СПЕЦИФИКАЦИИ | «Состав/Спецификация» (группы specSchema + цена/наличие) | Спецификация (варианты) |
| 4. Цена | Скил шаг4 + цены источников (select_pump/find_pump_by_sku) + курсы · гейт | «Смета» (закупка→себестоимость→наценка→цена) | Смета + цена |
| 5. Выход | Скил шаг5 (номенклатура→шифр) + НОРМЫ (валидация) | «Шапка» (шифр + цена) + документы | Шифр, расчётное дело |

Итоговая карточка (`cardLayout`) **собирается из выходов шагов** — каждый шаг
владеет своим блоком (уже правится в табе «Карточка»).

## Что снимает версия

```
TypeVersion (fire v3, status=published)
 ├─ шаги[]              (директивы + ссылки на скил-файлы версии)
 ├─ схема ввода          (FieldSpec)
 ├─ схема спецификации    (FieldSpec)
 ├─ дизайн карточки       (cardLayout)
 ├─ скил-файлы (КОПИИ)    (шаг1-5.md — свои у типа+версии, эталон цел)
 └─ нормы-ссылки          (какие СП/таблицы/якоря использует расчёт)
```
- **Источники подбора** — НЕ в версии типа: глобальный реестр `Source`
  (Wellmix/БД/сайты общие), шаг ссылается на MCP-инструменты.
- **Нормы** — глобальная библиотека; версия типа хранит ССЫЛКИ на нормы/якоря.
- Пайплайн гонит **published**-версию; правки → draft → «Опубликовать»; откат =
  поднять старую версию (скопировать её содержимое в новый draft).

## Модель БД (предлагается)

- `TypeVersion { id, typeCode, version Int, status: draft|published|archived, note, createdAt }`
  `@@unique([typeCode, version])`. Активная = единственная published.
- Привязать к версии (добавить `typeVersionId`):
  - `TypeStep.typeVersionId` (шаги — per-version).
  - Схема ввода: переиспользовать `TypeSchema` (уже версионируется) ИЛИ вложить в
    TypeVersion. Решить при реализации (склоняюсь: TypeSchema остаётся, TypeVersion
    ссылается на её version).
  - Спецификация/карточка: перенести `specSchema`/`cardLayout` с `SystemType` в
    `TypeVersion` (или в связанную запись).
- `TypeSkillFile { id, typeVersionId, path, content }` — КОПИИ скил-файлов типа.
  `@@unique([typeVersionId, path])`. Засев — копией из эталона при создании первой
  версии типа.
- `TypeVersionNorm { typeVersionId, normId, anchor? }` — ссылки на нормы (или Json
  на TypeVersion).

## Чтение скилов агентом (ключевой момент)

Сейчас агент читает `шаги/шагN.md` из `KIMI_AGENT_WORKSPACE` (эталон/volume).
Нужно: раннер **материализует скил-файлы published-версии типа** во временную
рабочую директорию агента (или отдаёт через MCP), чтобы агент читал КОПИИ ТИПА,
а не общий эталон. Варианты:
- (A) На старте прогона писать `TypeSkillFile[]` версии во временный `--add-dir`
  и указывать его агенту.
- (B) MCP-инструмент `read_type_skill(path)` — агент читает файл версии из БД.
Решить в Фазе 3 (склоняюсь к A — меньше меняет скил-механику).

## Фазы

1. **TypeVersion + скилы копией.** Модель TypeVersion + TypeSkillFile; при создании
   v1 типа — копия скил-файлов эталона в тип; UI: у типа видна версия, список
   версий, «черновик/опубликовать». Пилот fire.
2. **Раннер читает скилы версии.** Материализация TypeSkillFile published-версии в
   рабочую директорию агента; пайплайн гонит скилы ТИПА, не эталон.
3. **Шаги → версия.** `TypeStep.typeVersionId`; правки шагов (директива/порядок/
   гейты) — в draft-версии.
4. **Схема/спецификация/карточка → версия.** Перенести specSchema/cardLayout в
   версию; схему ввода связать с версией.
5. **Нормы-ссылки → версия.** Версия типа ссылается на нормы/якоря; шаги 2/5
   получают нужные СП из версии.
6. **Черновик/публикация/откат UI.** Полный цикл: draft → сравнение → publish →
   rollback. Гейты редактирования через версию.
7. **Приёмка: готовый пайплайн + скилы + тест на ≥3 проектах** (см. ниже).

## Готовый результат (Definition of Done)

Не «инфраструктура готова», а **работающий сквозной результат** на fire:

1. **Готовый пайплайн по всем шагам.** Published-версия fire проходит Вход →
   Расчёт → Подбор → Цена → Выход **end-to-end** без ручных костылей; каждый шаг
   даёт свой выход и свой блок карточки (характеристики / состав / смета / шифр).
2. **Скилы отредактированы под новые шаги** (КОПИИ в типе, эталон цел):
   - шаг3-подбор: источники по приоритету (БД → API `select_pump` → доверенные
     сайты `list_trusted_catalogs` → веб), заполнение по схеме спецификации;
   - шаг4-цена: цены из источников (реальные), потом оценка;
   - шаг2/5: нормы из версии (СП 8/10 таблицы), валидация;
   - тексты шагов согласованы с фактической механикой (не «модель не выбираем»,
     если API даёт модель с наличием).
3. **Протестировано на ≥3 реальных проектах** (ТЗ): для каждого — заполненная
   карточка ввода → характеристики → спецификация с ценой/наличием → смета →
   шифр; результат виден в карточке (cardLayout) и **сохранён в System**
   (Фаза A моста). Расхождения/провалы зафиксированы и исправлены правкой
   версии (демонстрация цикла draft→publish).
4. **Наглядно:** можно открыть 3 прогона и увидеть готовые ТКП — характеристики,
   состав, смета, шифр. Это и есть «готовый результат».

⚠ Тест на 3 проектах по подбору через Wellmix требует **рабочего токена**
(текущий — «Пользователь не существует»); до токена подбор идёт по БД-каталогу и
доверенным сайтам, API-ветка проверяется отдельно по готовности токена.
