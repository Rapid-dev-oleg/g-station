# Конструктор схем расчёта — декомпозиция и план

Цель: no-code инструмент, которым инженер/руководитель создаёт и ведёт **типы
расчёта** (схема ввода + инструкции + нормы/ГОСТ), чтобы тиражировать систему
с G-FIRE на G-WATER/G-POWER/G-DRAIN/G-CLEAN без правок кода. Прямо реализует
установку клиента «новый тип = данные, не код» и «живой итеративный ТЗ».

---

## Как это работает сейчас (по факту кода)

Архитектура «ядро + модуль типа»:

| Артефакт | Где | Роль |
|---|---|---|
| Реестр `SystemType` | БД (только сид, **UI нет**) | code, name, status, triggers, purposes, components, skillName, typeModule, schemaRef |
| Модуль `типы/<тип>.md` | файл в скиле (правится через «Методику») | 6 разделов контракта типа |
| `шаги/1–5*.md` | файл в скиле | **общий** конвейер для всех типов |
| `KNOWLEDGE/*` | файлы | правила-расчёта, инженерный-расчёт, нормативы, nomenclature, схема дела |
| `схема/расчётное-дело.schema.json` | файл | JSON-схема дела (**монолитная**, с fire-полями) |

**Контракт типа — 6 фиксированных разделов** (`типы/README.md`):
1. Идентификация (station_type, сегменты шифра, триггеры) — шаг 1
2. Нормативная база (СП/ГОСТ) — шаг 2
3. Опросный лист (обязательные поля карточки) — шаг 1
4. Нормативный расчёт (формулы целевых характеристик) — шаг 2
5. Особенности подбора (серия ШУ, материалы, спецоборудование) — шаг 3
6. Особенности оформления (форма ТП, спецразделы) — шаг 5

Добавить тип сейчас = вручную: INSERT `SystemType` (в сиде) + написать
`типы/<тип>.md` через файловый редактор. Нет: UI реестра типов, per-type схемы
ввода как данных, привязки норм к типу, тест-прогона.

### Что мешает тиражированию (болевые точки)
- **Схема дела монолитна**: `input`/`calc` содержат fire-специфику
  (`fire_params`, `fire_reserve_volume`). Для водоснабжения нужны другие поля.
- **Форма ревью (Гейт 1) захардкожена** под fire-карточку.
- **Валидация (ajv)** — по одной общей схеме, не per-type.
- Часть логики деления/карточки — в коде (`parse-document.ts`, `SPLIT_PRINCIPLE`),
  а не в данных типа.

## Что порождает конструктор (маппинг на существующее)

| Раздел контракта | Артефакт, который ведёт конструктор |
|---|---|
| 1. Идентификация | запись `SystemType` (triggers/purposes/components/шифр) |
| 2. Нормативная база | связи «тип ↔ Norm» (библиотека СП/ГОСТ) |
| 3. Опросный лист | **field-spec** типа (поля карточки как данные) → форма ревью + промпт агента + валидация |
| 4. Нормативный расчёт | инструкции шага 2 для типа (структура/markdown) |
| 5. Особенности подбора | номенклатура типа (серии ШУ, материалы, категории оборудования) |
| 6. Особенности оформления | инструкции шага 5 для типа |

Ядро (шаги 1–5, механика подбора по Q/H, формула цены, схема шифра) конструктор
НЕ трогает — оно общее.

## Модель сущностей (по видению НейроСофт)

```
CalcType (тип расчёта)
 ├─ идентификация: code, name, статус(draft/READY), серия/шифр, триггеры, назначения, компоненты
 ├─ Schema (схема ввода)  ── редактируется: добавлять/менять/удалять поля
 │    └─ Field[]  {ключ, подпись, ТИП ДАННЫХ, ед., опции, обяз., провенанс, подсказка,
 │                 значение по умолчанию, условие видимости}
 │        → из типа данных рисуется динамическая форма (заполнение/редактирование)
 ├─ Instruction[]  (создавать/прикреплять/редактировать) — разделы 4/5/6 контракта
 ├─ Normative[]    (создавать/прикреплять/редактировать) — раздел 2 (СП/ГОСТ), связь с Norm
 └─ SearchResource[]  ресурсы поиска оборудования, с приоритетом:
      • db   — запрос к каталогу (CatalogItem)
      • api  — эндпоинт поставщика/ПО подбора (URL + авторизация)
      • web  — веб-поиск по СПИСКУ ДОВЕРЕННЫХ САЙТОВ (домены + приоритет бренда)
```

**Решение по владению:** контейнер — `CalcType`; **Schema = поля ввода** (одна на
тип, версионируется); инструкции/нормативы/ресурсы вешаем на **тип**. Если
понадобятся несколько схем на тип — вынесем позже (открытый вопрос ниже).

### Словарь типов полей (для динамических форм)
Из текущей карточки уже нужны: `measured` (число + единица + провенанс
extracted/derived/assumed), `enum` (Select + опции), `boolean`, `number`, `text`/
`textarea`, `group` (вложенный объект, напр. `fire_params`), `array` (повторяемый
подсписок, напр. `reservoirs`). Плюс: `required`, `default`, `hint`, `visibleIf`
(условная видимость — напр. `jockey_Q` виден при `jockey_required=true`).
Рендер формы = маппинг «тип поля → виджет» (сейчас захардкожен в `IntakeFlow`).

### Ресурсы поиска (обобщение текущего хардкода)
Сейчас: `pricing/equipment.ts` (БД→веб) + `pricing/settings.ts`
(`brandPriority`, `manufacturerSites` — хардкод). Конструктор выносит это в
**данные на уровне типа**: упорядоченный список ресурсов `db | api | web` с
доверенными доменами и приоритетом. Питает фазу подбора/цен агента; связано с
клиентскими #10/#11 (доверенные источники, рейтинг) и #14 (веб-поиск).

## Декомпозиция (модули конструктора)

1. **Реестр типов — CRUD в UI.** Список типов (READY/PLANNED/draft), создать/
   клонировать/редактировать идентификацию: code, name, серия шифра, назначение,
   триггеры, компоненты. (Сейчас этого UI нет вообще.)
2. **Конструктор карточки (опросный лист как данные).** Per-type field-spec:
   поле = {ключ, подпись, ед., тип, обяз., провенанс, подсказка}. Ядро «схемы».
   Из него рендерится форма Гейта 1, собирается валидация, и спецификация полей
   инъектируется в промпт агента (парсинг/расчёт).
3. **Редактор инструкций типа** (разделы 4/5/6). Структурированные блоки +
   markdown с подсказками по контракту. Генерит/ведёт модуль типа.
4. **Библиотека норм/ГОСТ + привязка к типу** (раздел 2). Управление `Norm`
   (код/название/URL/резюме), пометка «доверенный источник» (#10, #11), связь
   норм с типом.
5. **Номенклатура типа** (раздел 5, #9). Категории оборудования, серии ШУ,
   материалы, правила шифра; производители и URL каталогов для авто-поиска.
6. **Тест-прогон типа.** Прогнать пример ТЗ через конвейер новым типом →
   карточка/расчёт + метрики (переиспользуем `verify-kimi`). Петля «живого ТЗ»
   (#16) и обучающих материалов (#8).
7. **Версии + активация.** draft → READY, версионирование модуля/схемы, откат.
   Тип виден в проде только в статусе READY.

## Решения

### Приняты (из видения НейроСофт)
- **Тип имеет редактируемую схему полей** с типами данных → динамические формы.
- **Инструкции/нормативы/ресурсы** — создаются/прикрепляются/редактируются, висят
  на типе. **Ресурсы поиска** = список `db | api | web (доверенные сайты)`.
- **Форма ревью** становится динамической (рендер из field-spec, не хардкод).

### Открыты — нужны до старта Фазы 1
1. **Источник истины типа: файлы `типы/*.md` или БД?**
   - (A) markdown-модуль источник + запись реестра (меньше работы, реиспользует монтирование методики).
   - (B) тип целиком в БД структурно; модуль/схему для агента **генерируем на лету**.
   - **Рекомендация: (B)** — под мультитенант, версии, тест-прогоны; но требует расшить схему дела.
2. **Область типов: общие для платформы или per-workspace?** (сейчас `SystemType`
   общий; логично: базовые общие, правки/новые — на воркспейс).
3. **Одна схема на тип или несколько?** Рекомендация: одна (версионируемая);
   несколько — если один тип имеет разные опросные листы по сценарию.
4. **`api`-ресурс** (ПО подбора/эндпоинт поставщика) — нужен сразу или позже?
   Сейчас реально есть только `db` + `web`; `api` можно заложить в модель, реализовать во Ф4.
5. **Расшивание монолитной схемы дела** (`расчётное-дело.schema.json`) на
   generic-ядро + per-type секцию — как именно, чтобы не сломать расчёт fire.

## Ответ по #5 — расшивание схемы дела (низкий риск)

Проверка кода: `validateDossier` = `ajv strict:false`, `StationInput` без
`required` и с `additionalProperties:true` → **валидация уже пропускает любые
поля input**. Жёсткость держат TS-типы + захардкоженная форма, не схема.
Хранилище `System.dossier` — JSONB, тоже без ограничений.

Поэтому **не «рвём» схему**, а вводим слой:
- **Стабильный конверт** (в коде): `meta`, `stations[]`, `calc`, `output`,
  `pricing` — общее для всех типов, не трогаем.
- **`input` остаётся гибким мешком**; его форму/валидацию/промпт задаёт
  **field-spec типа** (Schema из БД). Общие поля насосной станции (Q, H, схема,
  насосы, жокей, исполнение, питание) — базовый шаблон field-spec, который типы
  наследуют; type-специфика (fire_params/reservoirs у fire) — добавочные поля.
- **Валидация per-type** = стабильный конверт (ajv) + опциональная проверка
  `input` по field-spec (компилируем ajv из Schema). Для fire это воспроизводит
  текущее поведение (проверяем регрессом).

Итог: динамическими становятся **форма + промпт + (опц.) валидация**, а хранение
и конвейерная механика — прежние. Fire не ломается, новые типы добавляют поля.

## Фазы

**Фаза 1 — Фундамент «тип как данные».**
- UI реестра типов (CRUD `SystemType`).
- Вынести опросный лист в field-spec (данные); расшить схему дела на
  generic + per-type; динамическая форма ревью из field-spec.
- **Пилот: воспроизвести G-FIRE полностью через данные** (не сломав текущий расчёт).

**Фаза 2 — Инструкции и нормы.**
- Редактор 6 разделов модуля + генерация артефакта для агента.
- Библиотека норм/ГОСТ + привязка к типу (доверенные источники, #10/#11).

**Фаза 3 — Проверяемость.**
- Тест-прогон типа на примерах + метрики (`verify-kimi`); версии + активация.

**Фаза 4 — Номенклатура и обучение.**
- Номенклатура типа + производители/URL каталогов (#9); загрузка обучающих
  материалов (#8).

**Порядок доказательства:** сначала Фаза 1 с пилотом «G-FIRE как данные» — если
существующий тип полностью описывается конструктором и расчёт не деградирует
(сверка `verify-kimi` до/после), архитектура верна; дальше заводим G-WATER.

## Фаза 1 — задачи (решения: БД-источник, типы общие, одна версионная схема, api в модель)

**A. Данные и модель**
- **A1.** Модель БД: `TypeSchema { id, typeCode(FK SystemType), version, fields:Json, status(draft|active), createdAt }`; у типа — ссылка/выбор активной версии. TS-тип `Field { key, label, dataType, unit?, options?, required, default?, hint?, visibleIf?, provenance? }`. Миграция **аддитивная**. Значения `dataType`: `measured | number | enum | boolean | text | textarea | group | array`.
- **A2.** Field-spec **G-FIRE v1** — перенести текущие поля карточки (из `IntakeFlow`/`StationInput`) в данные `TypeSchema`; засеять как активную версию fire.
- **A3.** Server actions (права: супер-админ/руководитель): список типов, создать/редактировать идентификацию типа; `getSchema`, `saveDraftSchema`, `publishSchemaVersion`. Типы общие (без workspace-скоупа).

**B. Конструктор (UI)**
- **B1.** Реестр типов: список (статус, версия схемы) + форма идентификации (code, name, серия/шифр, триггеры, назначения, компоненты, статус).
- **B2.** Редактор схемы: список полей с drag-reorder; на поле — тип данных, ед., опции (для enum), обяз., default, hint, `visibleIf`; черновик → «Опубликовать версию».

**C. Динамическая форма и связка**
- **C1.** Компонент `DynamicForm(fields, value, onChange)` — маппинг `dataType → виджет` (measured→NumberInput+ед., enum→Select, boolean→toggle, text/textarea→Input, group→вложенная секция, array→повторяемый подсписок); поддержка `visibleIf`.
- **C2.** Встроить `DynamicForm` в ревью (`IntakeFlow`, `SystemWizard`, `CardSummary`) вместо хардкода — форма строится по активной схеме типа системы. Поведение для fire — идентичное.
- **C3.** Промпт агента (`parse-document`, `kimi-calc`) получает field-spec активной схемы типа как **опросный лист** (что извлекать/считать) — вместо зашитого в код списка.
- **C4.** (Опц.) Валидация per-type: компиляция ajv из field-spec поверх стабильного конверта.

**D. Пилот и регресс**
- **D1.** Доказать «данные, не код»: форма fire идентична прежней (глазами + смоук); `verify-kimi` на **5 выборочных кейсах** (по 1 на автора: `anohin-01, dasha-01, izosimova-01, nikitin-01, zagoryansky-01`) без деградации; завести **черновой 2-й тип** (заглушка G-WATER: пара своих полей) и убедиться, что форма/промпт строятся из данных без правок кода.

**Порядок исполнения:** A1→A2→A3 (фундамент, автономно) → B1→B2 (конструктор) →
C1→C2→C3 (динамика и связка) → D1 (пилот). A и B можно вести параллельно C1.

**Риск-контроль:** после C2/C3 — обязательный прогон `verify-kimi` на 5
выборочных кейсах (fire не должен просесть) и ручная сверка формы ревью.

## Риски
- Расшивание монолитной схемы дела — аккуратно, с регрессом `verify-kimi` на 45
  кейсах (fire не должен просесть).
- Валидация per-type вместо одной ajv-схемы.
- Логика в коде (`parse-document.ts`: деление на системы, сборка карточки) —
  часть контракта типа сейчас в коде; решить, что переносится в данные, что нет.
