# План: проект = ТЗ, мульти-типовой расчёт (организация существующего)

Статус: черновик, обсуждён с пользователем 23.07.2026. Пилот — **G-FIRE**
(макс. объём ~12 ТКП/мес, мин. вариативность, жёсткие таблицы СП 10.13130.2020 /
СП 8.13130.2009 / НПБ 88-2001*, прозрачная цепочка расход→давление→подбор→
спецификация→стоимость).

Цель — не переписать, а **организовать** уже существующие куски: парсер уже
мульти-типовой, `Project→System[]` уже есть; новый пайплайн-флоу это игнорирует.

## Единая модель (решения пользователя)

```
Project      = ТЗ (пакет файлов)
  └─ System  = один обнаруженный тип/подсистема к расчёту   (Project→System[] — уже есть)
       └─ PipelineRun = прогон расчёта этой System           (привязать systemId; уже поле есть)
```

- **Обнаруженный тип → сразу черновик `System`** под проектом (как `autoSubmit`
  в старом /intake уже делает: цикл по `parsed.systems[]`, по System на тип).
- **`PipelineRun` привязан к `System`** (`systemId`), по готовности `summary`
  пишется обратно в `System` (dossier + totalCost/clientPrice). System = durable
  запись, PipelineRun = журнал исполнения. Проект видит все расчёты.

## Что уже есть (переиспользуем)

- `parse-document.ts` → `ParsedDocument.systems[]` (массив, у каждой свой
  `typeCode`; тип по реестру `SystemType.triggers/purposes`, data-driven).
  `SPLIT_PRINCIPLE` + `buildTypeChecklists` + `buildIdentificationBlock`.
- `parse.ts` → `autoSubmit`: Project + цикл создания System на каждый
  `parsed.systems`, все в одном проекте. Форсится при `systems.length > 1`.
- `Project→System[]` (FK), `System.typeCode→SystemType` (FK), `DynamicForm`
  из `TypeSchema` (карточка по схеме типа), `PipelineRun.systemId?` (поле есть).
- В «Задачах» пайплайн-прогон уже кликабелен → `/calc/runs/[id]` (сделано 23.07).

## Разрывы (что чинить)

1. `parseToCard` (parse.ts) берёт только `systems[0]` — теряет остальные типы.
2. `/calc/new` (CalcWizard) — один тип руками; `PipelineRun` стартует без
   `systemId`/`projectId`; результат не пишется в System.
3. `/intake` создаёт Project+Systems, но не запускает пайплайн; `/calc/new`
   запускает пайплайн, но не создаёт Project/System. Мост отсутствует.
4. `commitIntake` / ручной review /intake зашиты на `typeCode:'fire'` и 1 систему.

## Фазы

### A. Мост PipelineRun ↔ System (фундамент, малый риск, аддитивно)
- `startPipeline`/`startPipelineRun` принимают `systemId`; проставляют его в
  `PipelineRun`. `projectId` берём через System.
- По завершении (в фон-хендлере после `summarizeRun`) пишем результат обратно в
  `System`: dossier/summary + `totalCost`/`clientPrice` из `summary.estimate`,
  статус System → рассчитана.
- На странице System показываем прогон/результат (`CardRenderer`) + ссылку на
  `/calc/runs/[id]`.
- **Ничего не ломает**: старые прогоны без systemId работают как есть.

### B. Проектный мульти-типовой приём (новый флоу поверх готового парсера)
- Точка входа: загрузка папки ТЗ → парсинг (уже мульти-тип) → `Project` +
  черновик `System` на каждый `parsed.systems[]` (обобщить/переиспользовать
  `autoSubmit`, без хардкода fire и без «только первая система»).
- Страница проекта = список обнаруженных систем/типов (бейдж типа, статус,
  «Рассчитать»). Это и есть «извлечённые типы, привязанные к проекту».
- Сценарий «загрузил папку в тип расчёта» = тот же парсинг, показываем все
  найденные типы, инженер выбирает, что считать.

### C. Расчёт системы = пайплайн (сшивка)
- Открыл System → карточка по схеме типа (`DynamicForm` из `TypeSchema`),
  предзаполнена распарсенным `input` → «Рассчитать» → `PipelineRun` с `systemId`.
- Несколько типов → несколько прогонов под проектом; вернуться можно из «Задач»
  и со страницы проекта/системы.
- Результат — на странице System (Фаза A) + в проекте.

### D. Нормы как таблицы (отдельный трек, для честного fire)
- Пробел: у `Norm` нет поля «полный текст/таблица» — только `summary` + якоря
  (key/label/value), у засеянных норм пусто → в редакторе «пустые тексты».
- Fire считается по таблицам СП 10/8, НПБ 88 → нормам нужно хранить таблицы
  (расширить `content`/якоря или добавить структуру таблиц) и наполнить.
- Не блокирует A–C; нужно для качества подбора/давления.

## Открытые вопросы
- Оставляем `/intake` для сравнения (память: да). Новый проектный флоу — поверх,
  не удаляя старый.
- Где именно предзаполнять карточку System из `parsed.input` (Фаза C): при
  создании черновика (Фаза B) или при открытии на расчёт.
- Что писать в `System` из summary: dossier целиком vs отдельное поле
  `pipelineSummary Json?` (аккуратнее, не мешает старому dossier-формату).
