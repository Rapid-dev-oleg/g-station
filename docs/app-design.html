01

Периметр и дерево систем

Автоматизируем расчёт ТКП по продукции собственного производства (префикс G). Пилот — пожаротушение.

Тип системыСерияТКП / мес*Приоритет
ПожаротушениеG-FIRE~121 · пилот
Хозпит-водоснабжениеG-WATER · G-POWER~92
КНС / ЛОСG-DRAIN · G-CLEAN~83
Итого~29

*Среднемесячно за апрель–июнь 2026 (данные клиента). G-FIRE выбран пилотом: наибольший объём, наименьшая вариативность, жёсткая нормативная база (СП 10.13130.2020, СП 8.13130.2009, НПБ 88-2001*) — параметры в таблицах, не в экспертном суждении.

Логика ТКП

расход → давление → подбор насоса по каталогу → спецификация → стоимость

Подход к ТЗ

Живой итеративный документ: первая версия на реальных запросах → фидбэк инженеров → правки. Не статичное ТЗ на старте.

Развитие

Пилот на G-FIRE → тираж на G-WATER/G-POWER → G-DRAIN/G-CLEAN. Новый тип = данные (реестр), не переписывание кода.

02

Сквозной пайплайн ТКП

Один конвейер для всех типов станций: 5 шагов = 3 этапа для клиента. Исполняет LLM-агент по скилу pump-station-calc. Экспертные развилки выносятся инженеру (супервизинг).

Этап 1 · Парсинг
1
Вход + карточка

Разбор пакета документов

Загрузка ТЗ/опросного/проекта «как есть». Извлечение текста и картинок, определение типа станции и сценария, сборка карточки параметров. Каждое число помечено провенансом: из документа / выведено / допущение.

in: docx · pdf · xlsx · txtout: карточка (Q, H, схема, жокей, исполнение, резервуары)
Гейт 1Инженер правит извлечённую карточку и допущения перед расчётом.
Этап 2 · Расчёт «что нужно»
2
Целевые характеристики

Расчёт по нормативам и формулам

Рабочая точка Q/H (запас 5–10 %), запас пожарной воды (объём резервуаров), схема резервирования и число насосов, диаметр коллектора, мощность двигателя, тип пуска.

норм: СП 10/8, НПБ 88out: требования к станции, независимые от бренда
Этап 3 · Подбор «чем закроем»
3
Спецификация

Подбор оборудования

Класс и конструктив насоса, жокей, коллектор, шкаф управления (ШУФ/ШУФС), корпус и резервуары. Сверка по каталогу/прайсу через MCP к БД. Точная модель и бренд — за инженером.

src: каталог · прайсы · веб-поискout: спецификация оборудования
4
Ценообразование

Себестоимость и цена

Расчёт себестоимости и клиентской цены по прайсам, курсу валют и наценке.

Гейт 2Инженер подтверждает курс, скидки, бренд насоса, коэффициент наценки.
5
Выход

ТКП, смета, валидация

Шифр изделия, техпредложение серии G-Fire, смета, чек-лист валидации. Результат — «расчётное дело» (единый JSON-контракт всех этапов).

out: расчётное дело · ТП · смета · PDF КП
Гейт 3Инженер проверяет черновик по чек-листу. Гейты схлопываются по мере роста качества.

04

Модель данных

Текущие сущности (Prisma) сгруппированы по назначению. есть — в схеме, план — предложение под задачи клиента.

Доступ
  • User · role (сейчас ENGINEER/ADMIN)
  • план Workspace — компания/воркспейс, tenantId на всех сущностях
  • план роли DIRECTOR · ENGINEER · MANAGER
Проекты и расчёт
  • Client · Project (status) · System
  • System.dossier — «расчётное дело» (JSONB, валид. ajv)
  • kimiCalc + hash — кеш расчёта агента; approvedSnapshot — заморозка утверждённого
  • план Lead / Deal — если часть воронки ведём у себя
Каталог и номенклатура
  • Manufacturer · ProductCategory
  • PriceList · CatalogItem (sku, attributes, price)
  • план URL каталогов производителя для авто-поиска
Нормы и правила
  • Norm (СП/ГОСТ: code, title, url)
  • RuleConfig — декларативные правила подбора (версионируемые)
  • план Source + рейтинг достоверности 1–10 (порог >7)
Типы систем · движок
  • SystemType — реестр типов (fire/water/…): триггеры, назначения, модуль скила
  • Job — фоновая задача (parse/calc), прогресс, результат
  • новый тип станции = строка в реестре + модуль-типа в скиле
Настройки
  • Settings (singleton) — реквизиты, курсы, наценка, ключи, бэкенд агента
  • план вынести из singleton → на воркспейс (мультитенант)
  • план дашборд параметров воронки (порог скоринга, квалификация)
05

Роли и мультикомпанийность

Фундамент, который клиент просит заложить сразу. Внутри компании — фактически две компании с раздельными данными; перспектива — продавать продукт другим.

Мультитенантность · открытый вопрос

Уровень изоляции нужно решить до наращивания данных:

  • Воркспейс + row-level в одной БД (tenantId на сущностях) — проще эксплуатировать, дешевле масштабировать на новых клиентов.
  • Отдельная БД на компанию — сильнее изоляция, дороже в обслуживании.

Рекомендация: воркспейс + row-level, заложить tenantId с самого начала (ретрофит дороже).

06

Встраивание в воронку продаж

Битрикс24 — система воронки; наш модуль встраивает AI по стадиям. Стадия 3 (ТКП) — стык с ядром расчёта и фокус пилота. Остальное — фаза 3.

1

Новая

Автоквалификация заявки (тип/объём/регион/бюджет), отсев нецелевых с тендерных площадок, маршрутизация, автоназначение менеджера, автозаполнение карточки из ЕГРЮЛ/тендеров.

2

Сбор информации · выход на ЛПР

Автопоиск контактов ЛПР, обогащение карточки (ИНН → отрасль, тендеры, аффилированные), AI-бот первичного сбора техданных.

3

Подготовка КП ядро расчёта

Автогенерация ТКП (подбор оборудования, спецификация, стоимость), актуальные цены/сроки поставщиков, проверка на нормы (СП/СНиП/ГОСТ), генерация PDF КП.

4

КП отправлено · согласование

Автоотправка КП, follow-up (день 3/7/14), AI-анализ ответа (согласовано/правки/отказ), автофиксация статуса.

5

Согласие · документы

Автогенерация договора, пакет (договор, счёт на аванс, спецификация), маршрутизация на согласование, контроль аванса.

6

Договор подписан · реализация

Задачи производству по авансу, заявка на закупку из спецификации, уведомления о статусе, УПД.

7

Отгружено

Уведомление об отгрузке, NPS через 2 недели, отчёт по проекту, AI-классификация рекламаций.

∞

Сквозной AI-контроль

Скоринг сделки, анализ звонков (транскрипция/тональность) и переписки, контроль активности (брошенные сделки), вероятность закрытия, автоприоритизация портфеля.

08

Что оптимизировать в структуре

Решения, снижающие сложность и удешевляющие масштаб на другие типы систем.

A

Один путь расчёта — LLM

Убрать TS-движок (src/lib/engine + потребители в UI): сейчас два параллельных расчёта дублируют методику. Останется единый Kimi-путь. Открыто: чем заменить регрессию качества (npm run verify, 49 кейсов) — эталонным прогоном с метриками.

B

Единый контракт «расчётное дело»

Держать System.dossier (JSONB + JSON-схема) единственным объектом, который течёт через все этапы и гейты. Уже так — закрепить как правило, не плодить параллельных форматов.

C

Типы систем — данными, не кодом

Реестр SystemType + модуль-типа скила: добавление G-WATER/G-DRAIN = строка реестра + md-модуль. Ядро конвейера не трогаем — так тираж на новые типы дешёвый.

D

Мультитенантность с первого дня

Заложить tenantId/воркспейс на все сущности сразу и вынести Settings из singleton. Ретрофит изоляции позже — дорого и рискованно.

E

Единая абстракция источников

Норма / каталог / веб-поиск — один интерфейс «доверенный источник» с рейтингом 1–10 и приоритетом. Переиспользуется в проверке норм (#10), скоринге источников (#11) и веб-поиске оборудования (#14).

F

Чёткая граница с Битрикс24

Воронка и коммуникации живут в Битриксе; наш модуль отвечает за ТКП (стадия 3), номенклатуру и расчёт. Синхронизация — через API-прокладку. Не дублировать CRM у себя без нужды.

G

Гейты как конфиг, а не хардкод

Супервизинг-проверки — настраиваемые чекпоинты, которые схлопываются по мере роста качества расчёта. Один механизм на все типы, управляется из настроек.
