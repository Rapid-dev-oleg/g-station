# TS-движок расчёта станции — НЕ ИСПОЛЬЗУЕТСЯ В UI

**Статус: тестовый стенд (testbed), не прод.** Расчёт в боевом UI идёт через
Kimi-агента по скилу `pump-station-calc` (см. `src/server/actions/kimi-calc.ts`).

## Зачем тогда этот код существует

1. **Дисциплина правил.** Здесь декларативно записаны 12-зонная матрица
   класса насоса (`calc/pump-class.ts`), карта аналогов брендов
   (`calc/brand-map.ts`), правила DN коллектора (`calc/collector-dn.ts`).
   Это **эталонная** реализация правил из `gidrostroy/KNOWLEDGE/правила-расчёта.md` —
   её можно проверять `npm run verify` против 49 эталонных кейсов.
2. **Baseline для будущего гибрида.** Если решим возвращать расчёт на TS
   (быстрее, детерминированно), вся методика уже есть в коде.
3. **Регрессионный контроль методики.** При правках правил-расчёта.md
   движок показывает на скольких из 49 кейсов методика «совпала».

## Что НЕ делать

- Не звать `runSystemCalc` / `calcPipeline` из UI-server-actions.
- Не рендерить `CalcPanel` (старый компонент) на маршрутах UI.
- Не добавлять сюда фичи, нужные только продакшен-расчёту — они должны идти
  в Kimi-пути (`kimi-calc.ts` + `pricing/processor.ts`).

Промахи класса разделены по авторам ТЗ (Анохин → MULTISTAGE, Загорянский/

по физике.

## Когда удалять

Решение об удалении — после того как Kimi-путь стабильно даст ≥ 90 % по
аналогичным метрикам на тех же 49 кейсах. До тех пор движок остаётся как
страховка и контрольный стенд.
