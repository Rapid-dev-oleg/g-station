# g-station — дизайн приложения (инструкция для ИИ)

Документ для ИИ-агента, который разрабатывает или поддерживает g-station.
Описывает НАЗНАЧЕНИЕ, процесс, экраны, стиль и принципы. Опирайся на него,
не изобретай заново и не противоречь ему.

## 1. Что это и для кого

Веб-приложение для инженера компании «Гидрострой-НН»: по техническому
заданию клиента подобрать **насосную станцию** (пожаротушение / водоснабжение)
и подготовить **ТКП** (техническое предложение + смета).

Пользователь — инженер-расчётчик. Он НЕ программист и НЕ хочет вбивать
десятки полей вручную.

## 2. Ключевой принцип (САМОЕ ВАЖНОЕ)

**Приложение — ассистент, а не форма ручного ввода.**
- Тяжёлую работу делает **LLM (Kimi)**: читает ТЗ (даже скан), извлекает
  параметры, считает станцию по методике (скил `pump-station-calc`).
- Инженер только **проверяет на гейтах и утверждает**. Он редактирует то,
  что Kimi уже извлёк/посчитал, а не заполняет с нуля.
- Расчёт идёт через LLM по инструкции-скилу, НЕ через захардкоженный
  TS-движок (движок остался только для регресс-теста `npm run verify`).

## 3. Доменная иерархия

```
Клиент → Проект (объект) → Система (1..N, каждая своего типа)
```
- Один объект (проект) может иметь **несколько систем** (пожарная +
  хоз-питьевая). Kimi сам делит ТЗ на системы и определяет тип каждой.
- Тип системы (`fire` / `water`) задаёт правила расчёта.

## 4. Линейный процесс (маршрут инженера)

```
Проект ──«Загрузить ТЗ»──► Kimi читает → создаёт систему(ы) [статус: Вход]
   │
   ▼  список систем с цветными статусами
Система — ЕДИНЫЙ степпер из 3 шагов:
   1. ВХОД        компактная карточка распознанного, инженер правит      (гейт 1)
   2. РАСЧЁТ      Kimi считает по скилу → таблица пункт-значение-обоснование (гейт 2)
   3. УТВЕРЖДЕНИЕ снапшот замораживает данные                            (гейт 3)
   │
   ▼
Проект → «Сформировать ТКП» (ТП + смета из утверждённого снапшота)
```
Статусы системы: `INPUT` (вход) → `CALCULATED` (рассчитана) →
`FINALIZED` (утверждена). Переход — сайд-эффект кнопок степпера, отдельного
селектора статуса НЕТ.

Гейты инженера (методика их НЕ закрывает сама — помечать «на проверку»):
точная модель насоса, бренд/производитель, коэффициент наценки.

## 5. Экраны

- **Дашборд** — KPI портфеля.
- **Проект** (`/projects/[id]`): карточка объекта, сводка, список систем с
  цветными статусами; кнопки «Загрузить ТЗ» (→ Kimi), «Система вручную»,
  «ТКП». Одна кнопка на систему — «Открыть».
- **Система** (`/projects/[id]/systems/[sid]`): единый степпер (см. §4).
  НЕТ отдельной страницы расчёта (`/calc` редиректит сюда).
- **Каталог** — оборудование из прайсов (для будущего подбора артикулов).
- **Настройки** — реквизиты, ключ Kimi (`kimiKey`), ключ OpenRouter.

## 6. Стиль — функциональный минимализм

- Просто, понятно, точно + **вариативность** где уместно.
- **Результаты расчёта — таблицей**, не сырым текстом LLM. Формат строки:
  `параметр — значение (редактируемо) — обоснование`. Спорное (гейты) —
  подсвечено «на проверку инженеру». Подробный текст LLM — под кнопку.
- **Карточка-вход — сгруппирована** по смыслу (Гидравлика / Назначение /
  Исполнение / Резервуары). Цель: **быстрые кнопки-чипсы** для типовых
  значений (схема 1/1·2/1·2/2, назначение, материал) вместо длинных
  дропдаунов. (план — ещё не везде сделано.)
- Никаких вложенных степперов, дублирующих экранов, кучи кнопок без смысла.

## 7. Как встроен Kimi

- **Парсинг ТЗ**: документ → если скан/картинки, извлекаем изображения
  (`pdftoppm` для PDF, `unzip word/media` для docx) → **Kimi vision**
  (`kimi-for-coding`) извлекает карточку (`parse-document.ts`). Текстовые
  документы — текстом.
- **Расчёт**: **Kimi CLI агент** (`runKimiAgent`, subprocess) считает по
  скилу `pump-station-calc` (методика в `gidrostroy/.claude/skills/` +
  `KNOWLEDGE/`), смонтированной как workspace. Агент возвращает текст;
  **второй дешёвый chat-запрос структурирует** его в JSON
  `items[{param,value,rationale,gate}]` + `code` (агент сам JSON не отдаёт
  стабильно). Результат кешируется в `System.kimiCalc` по хешу карточки.
- **Endpoint**: `api.kimi.com/coding/v1`, заголовок `User-Agent: claude-code`,
  `thinking: disabled` (иначе reasoning съедает бюджет). Ключ из настроек
  (`Settings.kimiKey`) → `MOONSHOT_API_KEY`.

## 8. Что НЕ делать

- НЕ латать старый код поверх (не оборачивать старый визард в новый степпер).
  Сначала смотреть схему процесса (`gidrostroy/KNOWLEDGE/процесс-расчёта.md`),
  переделывать от сути.
- НЕ делать форму ручного ввода 6+ шагов — ввод от Kimi, инженер правит.
- НЕ показывать сырой markdown LLM как результат — структурировать в таблицу.
- НЕ считать станцию TS-движком в UI — только через Kimi-скил.
- НЕ запускать `npm run build` при работающем `npm run dev` (делят `.next`,
  ломается статика 404).

## 9. Стек и ключевые модули

- Next.js 15 (App Router) + React 19 + TypeScript strict, Postgres + Prisma,
  next-auth, CSS Modules (без Tailwind). Деплой — Docker (см. `DEPLOY.md`).
- `src/server/ai/parse-document.ts` — парсинг ТЗ в системы.
- `src/server/ai/kimi-agent.ts` / `kimi.ts` — запуск Kimi (агент / chat).
- `src/server/actions/kimi-calc.ts` — расчёт + структуризация + кеш.
- `src/server/actions/approve.ts` — утверждение (снапшот).
- `src/components/system/SystemFlow.tsx` — степпер системы.
- `src/components/system/CardSummary.tsx` — карточка-вход.
- `src/components/calc/KimiCalcPanel.tsx` — таблица расчёта.
- `prisma/schema.prisma` — System: `status`, `dossier`, `kimiCalc`,
  `kimiCalcHash`, `approvedSnapshot`. Правила расчёта — `RuleConfig` (БД).
